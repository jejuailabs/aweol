# 애월초 학급 전시실 — 작업 상태

> 새 세션에서 이어서 작업할 때 이 문서만 읽으면 된다. 마지막 갱신: 틀린그림 찾기 완료 시점.

## 무엇을 만들고 있나

초등학교 학급 작품을 **3D 가상 공간**에 전시하는 웹앱. 지도에서 학교를 고르면
3D 학교 → 교실 → 전시실로 들어가며, 아바타가 걸어다니며 작품을 관람한다.
비로그인 관람은 완전 자유이고, 로그인은 필요한 사람(교사·학생·학부모)만 한다.

- 배포: https://aweol.vercel.app (main 푸시 → Vercel 자동 배포)
- 저장소: github.com/jejuailabs/aweol (**jejuailabs 계정으로 푸시**. 로컬에 다른 깃허브 계정도 있으니 주의)

## 지켜야 할 규칙 (어기면 터진 적 있음)

1. **`firebase-admin/auth` 를 import 하지 말 것.**
   `jwks-rsa@4 → jose@6`(ESM)가 서버리스에서 `require() of ES Module` 로 터져
   해당 API 전체가 500이 된다. 토큰 검증은 `lib/firebase-admin.ts` 가 jose로 직접 한다.
   Firestore 쪽(`firebase-admin/firestore`)은 안전하다.

   **2026-07-23 재확인 — 아직도 유효하다.** 확인용 라우트를 배포해 직접 재봤다:
   ```
   배포본(Vercel, Node v24.18.0): importedAuth=false
   ERR_REQUIRE_ESM: require() of ES Module
     /var/task/node_modules/jose/dist/webapi/index.js
     from /var/task/node_modules/jwks-rsa/src/utils.js not supported
   ```
   Node 22.12+ 부터 `require(ESM)` 이 정식 지원이라 이제 괜찮지 않을까 싶었는데 아니다.
   **로컬 프로덕션 빌드(Node 22.19)에서는 통과하고 배포본에서만 터진다** —
   로컬에서는 Turbopack 이 번들에 넣어버리고, Vercel 에서는 firebase-admin 을
   **외부 모듈로 두고 런타임에 raw `require`** 하기 때문이다
   (오류 메시지의 `Failed to load external module` 이 그 증거).
   그래서 Node 를 올려도 해결되지 않는다. `jose@6` 은 `exports.require` 가 아예 없다.

   **커스텀 토큰(아이 로그인)도 이 규칙에 걸린다.** `createCustomToken` 이
   `firebase-admin/auth` 에 있기 때문이다. 커스텀 토큰은 결국 서비스 계정 키로 서명한
   JWT 하나라, `lib/firebase-admin.ts` 의 `createStudentToken` 이 jose 로 직접 만든다.
2. **`server-only` 을 Route Handler 경로에서 import 하지 말 것.** 같은 이유로 500.
3. **로컬 통과 = 배포 통과가 아니다.** 위 두 문제 모두 로컬 dev에서는 재현되지 않았다.
   서버 API를 건드렸으면 반드시 배포 후 프로덕션에서 검증한다:
   ```
   $env:BASE_URL="https://aweol.vercel.app"; node scripts/verify-student-code.mjs
   $env:BASE_URL="https://aweol.vercel.app"; node scripts/verify-blackboard.mjs
   $env:BASE_URL="https://aweol.vercel.app"; node scripts/verify-homework.mjs
   $env:BASE_URL="https://aweol.vercel.app"; node scripts/verify-shop.mjs
   $env:BASE_URL="https://aweol.vercel.app"; node scripts/verify-role.mjs
   $env:BASE_URL="https://aweol.vercel.app"; node scripts/verify-quiz.mjs
   $env:BASE_URL="https://aweol.vercel.app"; node scripts/verify-school-scope.mjs
   $env:BASE_URL="https://aweol.vercel.app"; node scripts/verify-spot.mjs
   ```
   푸시 직후 바로 돌리면 구버전이 응답한다. 2~3분 기다렸다가 검증한다.
4. **Firestore는 중첩 배열을 저장 못 한다.** 좌표는 `[x,y,x,y,...]` 로 펴서 넣는다.
5. **이미지 생성 모델명은 임의로 바꾸지 말 것.** `gpt-image-2`, quality `low`.
   생성 이미지는 항상 Firebase Storage에 올리고 배포물에 포함하지 않는다.
   - `public/` 은 비어 있어야 한다. 여기 넣은 건 전부 Vercel 배포 용량이 된다.
   - **dataURL(base64)을 Firestore 문서에 저장하지 말 것.** 읽을 때마다 그 용량을 다시 낸다.
     이미지는 Storage에 올리고 문서에는 URL만 넣는다.
   - 이미지를 **교체**하는 경로에서는 옛 파일을 지운다. 안 지우면 바꿀 때마다 쌓인다.
     문서를 지우는 경로에서도 딸린 이미지를 함께 지운다.
   - **Firestore 읽기 횟수도 비용이다.** 문서 단위 과금이라 `getDocs` 한 번이
     그 컬렉션 문서 수만큼의 읽기다. 무료 한도 하루 5만 건.
     - **반 → 활동 → 작품처럼 중첩해서 도는 조회를 만들지 말 것.** 반이 늘면 곱으로 커진다.
       필요한 것만 `collectionGroup` + `where` 로 한 번에 가져온다
       (단, collectionGroup 은 중첩 규칙이 적용되지 않아 `match /{path=**}/...` 를 따로 열어야 한다).
     - 전체 집계가 꼭 필요하지 않으면 **펼칠 때 읽는다**(`/admin/[schoolId]` 의 작품 수).
     - 점검: `node scripts/audit-reads.mjs` — 화면별 읽기 수와 규모별 추정치.
     - 남은 한계: `/admin` 학교 목록은 학교·반·활동을 전부 읽어 50개교면 약 4,850건이다.
       학교가 많아지면 학교 문서에 집계 필드를 두는 쪽으로 바꿔야 한다.
   - 점검: `node scripts/audit-storage.mjs` (사용량·고아 파일·문서 속 base64).
     `--clean` 으로 고아 파일 회수. **`app-assets/schools/` 를 제외한 `app-assets/*` 는
     코드가 주소를 박아 쓰는 파일**(`lib/image-urls.ts`)이라 고아로 잡으면 안 된다.
6. **경로 문자열을 직접 조립하지 말 것.** `lib/paths.ts` 헬퍼를 쓴다.
7. 규칙 변경 후에는 `node scripts/deploy-firebase-rules.mjs` 로 배포한다.
   (Firebase CLI 계정에는 이 프로젝트 권한이 없어 서비스 계정으로 직접 배포하는 스크립트다)

## 현재 구조

```
/                                                  지도 (메인)
/school/[schoolId]                                 3D 학교
/school/[schoolId]/class/[classId]                 학급 정보
/school/[schoolId]/class/[classId]/room            3D 교실 (칠판·알림판·게시판)
/school/[schoolId]/class/[classId]/activity/[id]   3D 전시실
/admin/[schoolId]                                  학교별 관리 대시보드
/admin/[schoolId]/{approval,roster,class/[id]}
/admin/logs                                        접근 기록 (슈퍼관리자 전용)
/join-class                                        학생코드 입력
API: /api/{school,school-image,student-code,blackboard,homework,enhance,
          role,shop,quiz,quiz-explain,spot-game,spot-generate}
```

Firestore: `schools/{schoolId}/classes/{classId}/{students,activities,notices,homeworks,quizzes,spotGames,blackboard}`
- `studentCodes`, `blackboard`, `homeworks/*/submissions`, `quizzes/*` 하위,
  `users/*/{inventory,stampLedger}` 는 **클라이언트 쓰기 금지, 서버 전용**
- `quizzes/*/answerKeys`, `spotGames/*/answerKey` 는 **교직원만 읽기**
  (학생이 읽으면 퀴즈·놀이가 성립하지 않는다)
- users 문서의 `role`·`pendingRole`·`classIds`·`stamps`·`avatarCustom` 은 클라이언트 쓰기 금지
- `accessLogs` 는 슈퍼관리자만 읽기

### 숙제 읽음 확인 (2026-07-21)

**'아직 안 냈다' 와 '아예 못 봤다' 는 선생님에게 전혀 다른 이야기다.**
앞은 재촉할 일이고, 뒤는 알림이 안 닿았다는 뜻이라 연락 방법을 바꿔야 한다.
지금까지는 둘 다 '미제출' 한 칸이었다.

- 현황판이 다섯 칸: **안 봄 / 봤는데 안 냄 / 제출 / 검사완료 / 미연결**
- 콕 찌르기 아래에 "이 중 N명은 열어보지도 않았어요" 를 붙였다.
- 기록은 `homeworks/*/reads/{uid}`, **한 번만 쓴다.**
  규칙에서 `create` 만 허용하고 `update` 를 막아 '처음 언제 봤나' 가 남는다.
  열 때마다 덮으면 아이가 스무 번 열면 스무 번 쓴다.
- 읽음은 아이들끼리 못 본다 — 본인과 담임만.
- 교직원이 연 것은 안 남긴다. 선생님이 자기 숙제를 연 걸 셀 이유가 없다.
- 검증: `node scripts/verify-homework-read.mjs`

### 숙제 제출 종류 (2026-07-20)

선생님이 **하나를 고르면 아이 화면에는 그 입력창 하나만** 나온다.
여러 개를 열어두면 아이가 뭘로 내야 하는지부터 헷갈린다.
글쓰기 · 손글씨/그리기 · 사진 · **동영상 · 영상 주소** 다섯 가지.

**동영상은 두 갈래로 나눴다. 용량 때문이다** (실측 계산):
- `link` (유튜브 주소) — 저장 0, 트래픽 0. 긴 영상은 이쪽.
- `video` (파일 업로드) — 30초짜리도 한 반 25명이면 450MB.
  무료 5GB 로 숙제 11개면 찬다. 트래픽은 더 심해서 숙제 1개당 11GB(25명이 서로 보면),
  무료 100GB/월로 숙제 9개다. 그래서 **20MB 상한**을 걸고
  출제 화면에서 선생님에게 이 숫자를 그대로 알려준다.
- 상한은 `storage.rules` 와 `HomeworkPanel` 의 `VIDEO_MAX_MB` **양쪽에 있다.**
  규칙만 있으면 아이가 한참 올린 뒤 알 수 없는 오류를 보고, 화면만 있으면 안 막힌다.
  파일을 **고르는 순간** 알려준다.
- **붙여넣은 주소는 서버가 검사한다.** `javascript:` 나 `data:` 를 그대로 두면
  아이가 낸 주소가 반 전체가 누르는 함정이 된다. http/https 만 통과시킨다.
  화면에서도 `rel="noreferrer noopener"` 를 뺄 수 없다.

같이 메운 구멍:
- **마감일** — `dueDate` 가 스키마에는 있는데 `null` 로만 저장되고 입력칸이 없었다.
  이제 출제할 때 정하고, 목록에서 바로 배지로 보인다(지나면 빨강).
- **숙제 고치기** — 퀴즈에는 있는데 숙제에는 지우기만 있었다.
  제출 종류를 바꿔도 이미 낸 것은 지우지 않는다(아이가 한 걸 앱이 없애면 안 된다).
  대신 선생님에게 미리 알린다.
- 교사 현황판도 영상·링크를 보여준다. 안 고치면 멀쩡한 제출이 '내용이 비어 있어요'로 보인다.
- 검증: `BASE_URL=... node scripts/verify-homework-media.mjs`

**아직 없는 것**(다했니?다했어요 대비): 읽음 확인(공지를 봤는지),
출석부, 학생별 누가기록, 미제출 푸시 알림(지금은 콕찌르기만).

### 숙제 화면 구조 (2026-07-20 재설계)

교사 화면의 기준은 제출물이 아니라 **명부**다. `HomeworkTeacherGrid.tsx` 가
`students` 를 깔고 `submissions`/`nudges` 를 얹어 색을 칠한다.

- 3색: 미제출(베이지) / 제출(파랑) / 검사완료(초록). 학생코드 **미연결자는 점선 칸**으로
  따로 뺀다 — 제출 자체가 불가능한 상태라 미제출과 섞으면 헛되이 찌르게 된다.
- `checked` 는 재제출 시 초기화된다. 옛 내용을 보고 검사한 셈이 되면 안 된다.
- 콕 찌르기: `homeworks/*/nudges/{uid}`. 서버만 쓰고 읽기는 **찔린 본인과 교직원만**
  (친구가 누가 안 냈는지 훑어볼 수 없어야 한다). 제출하면 자동 해제.
- **반려 기능은 넣지 않기로 확정.** 오프라인 수업이 섞여 있어 알림 없이 반려하면
  아이가 영문을 모른다. AI 검수도 거부가 아니라 보류(교사 최종 판단), 코멘트로 갈음.
- 도장 지급은 이 화면의 **검사완료 버튼**에 붙어 있다 (아래 도장 경제 참고).

### 도장(쿠키) 경제 (2026-07-20)

버는 곳(숙제) → 쓰는 곳(상점) → 보이는 곳(3D 아바타)이 한 줄로 이어져 있다.

- **교사가 도장을 찍는 행위 = 검사완료 = 아이에게 도장 1개**. 셋을 하나로 묶어 손이 한 번만 간다.
  도안(참 잘했어요·고마워요·감사해요·최고예요)은 상점의 `stamp` 카테고리에서 챙기고,
  **보유한 것만** 찍을 수 있다. 지금은 전부 무료.
- **경제 관련 쓰기는 전부 서버(`/api/shop`, `/api/homework`)에만 있다.**
  규칙이 users 문서의 `stamps`·`avatarCustom` 변경을 막는다 — 열려 있으면
  자기 문서에 `stamps: 9999` 를 써넣고 상점을 털 수 있다.
  가격도 요청 본문이 아니라 `lib/shop-catalog.ts` 에서 읽는다. 안 그러면 0원에 사간다.
- 규칙에서 `get(키, 기본값)` 을 쓴 이유: 이 기능 이전 계정에는 `stamps` 필드가 없고,
  없는 필드를 그냥 참조하면 **규칙 평가가 실패해 아바타 선택·설정 저장까지 막힌다.**
- **도장 파밍 구멍 두 개**를 막아뒀다. 건드릴 때 주의:
  (1) 재검사 중복 지급 → `submission.awarded` 플래그.
  (2) 제출→검사→재제출→재검사 → 재제출이 문서를 통째로 덮어쓰며 `awarded` 를 흘리던 것을
  이어받게 했다. **submissions doc 은 set(merge)가 아니라 통째 덮어쓰기라 필드 추가 시 조심.**
- 검사를 취소해도 이미 준 도장은 회수하지 않는다. 받았다 뺏기면 아이가 상처받는다.
- 아이템은 `walker.tsx` 가 실제로 그릴 수 있는 것만 판다. 살 수는 있는데 껴도 안 보이면
  아이 입장에서는 도장을 버린 셈이 된다. 새 품목을 넣으려면 3D 파츠를 먼저 만든다.
- 유저 문서는 `auth-context` 에서 `onSnapshot` 으로 구독한다. 한 번만 읽으면
  받은 도장과 착용한 아이템이 새로고침 전까지 안 보인다.

### 역할과 승인 (2026-07-20)

- **role 은 서버(`/api/role`)만 정한다.** 규칙에서 `role`·`pendingRole`·`classIds` 의
  클라이언트 쓰기를 막았다. update 뿐 아니라 **create 도 막아야 한다** —
  첫 로그인에 `role: 'teacher'` 로 문서를 만들어버리면 그만이기 때문이다.
- 학생·학부모는 종전대로 즉시 부여(어차피 학생코드가 없으면 반에 못 들어간다).
  **교사만 `pendingRole` 로 접수되고 슈퍼관리자가 `/admin/teachers` 에서 승인**해야 role 이 된다.
- **교사 권한은 소속 학교 안에서만 통한다.** 신청할 때 학교를 고르고(`pendingSchoolId`),
  승인되면 `schoolIds` 에 그 학교만 들어간다. 규칙은 `isStaffOf(schoolId)` 로 판정하고
  API 는 `isStaffOfSchool(user, schoolId)` 을 쓴다. 전역 `isStaff()` 는 학교를 특정할 수 없는
  자리(collectionGroup)에서만 쓴다.
- `users` 문서 읽기는 본인과 총관리자만이다. 교사에게 열면 남의 학교 아이·학부모까지 보인다.
- `super_admin` 은 신청 대상이 아니다. Firestore 콘솔에서 직접 지정한다.

### 학교별 3D 외관 (2026-07-20)

건물 형태는 학교 공용이고, **이름·현판·색**만 학교별로 바뀐다.

- 간판 문구는 반드시 학교 문서의 `name` 에서 온다. 예전엔 '애월초등학교' 가 박혀 있어

- 대표 이미지는 업로드 시점에 가로 1024 / JPEG 로 줄인다(`lib/image-compress.ts`).

### 퀴즈 (2026-07-20)

유형 3종(객관식·단답형·서술형) + 자료 2종(이미지·유튜브) + AI 해설.

- **정답은 절대 클라이언트로 내려가지 않는다.** 문항(`questions`)과 정답(`answerKeys`)을
  다른 컬렉션에 나누고 `answerKeys` 는 교직원만 읽게 막았다. 문항 문서에 정답을 같이 넣으면
  개발자도구로 그냥 보인다. **채점도 서버에서 한다** — 클라이언트 채점은 결국 정답을 받아야 한다.
- 단답형은 공백·문장부호·전각을 정규화해 채점한다(`lib/quiz-utils.ts`).
  "3 개", "3개.", "３개" 를 다르게 보면 아는 아이가 틀린 것으로 나온다.
  허용 표기를 쉼표로 여러 개 받는다.
- **서술형은 채점하지 않는다.** 초등학생 글을 기계가 맞다/틀리다 하면 안 된다. 교사가 읽는다.
- AI 해설(`/api/quiz-explain`)은 **제출한 사람만** 부를 수 있다.
  안 그러면 풀기 전에 해설을 열어 정답을 알아낸다.
  한 번 만들면 문항에 캐시해 반 전체가 재사용한다 — 25명이 각자 부르면 같은 답을 25번 산다.
  교사가 직접 쓴 해설이 언제나 우선. 모델은 `OPENAI_TEXT_MODEL`(기본 gpt-4o-mini).
- 출제는 전부 검사한 뒤 배치 저장. 하나라도 잘못되면 통째로 거부해 반쪽 퀴즈를 남기지 않는다.
- 유튜브는 전체 URL이 아니라 **id 만** 저장한다(공유·짧은 링크·임베드·Shorts·live 모두 파싱).
- 점수는 아이에게 보여주지 않는다. 문항별 ⭕❌ 와 해설만 준다.

### 틀린그림 찾기 (2026-07-20)

알림판 다섯 번째 칸. 사진 하나로 만들고, 아이는 다른 곳을 찾아 누른다.

- **AI가 정확히 몇 군데를 바꿔줄지는 알 수 없다.** 그래서 `/api/spot-generate` 결과를

- 정답 좌표는 `spotGames/*/answerKey` 에 따로 두고 교직원만 읽는다.
  찍을 때마다 서버가 판정하고 맞은 자리만 돌려준다 — 좌표를 내려주면 1초 만에 끝난다.
- **시간은 서버가 잰다.** 클라이언트가 보내는 초를 믿으면 순위표가 의미 없다.

- 찾을 개수(`spotCount`)는 공개한다. 몇 개 남았는지는 알아야 한다. 좌표만 숨긴다.
- 세로로 긴 사진은 좌우로, 아니면 위아래로 배치한다(`layout`).
  두 그림은 같은 크기·방향으로 맞춰 저장한다 — 안 그러면 좌표가 어긋난다.
- 아직 없는 것: 제보 게시판, 댓글.

### 이미지 로딩 (2026-07-20)

- 작품은 업로드할 때 **썸네일(640px)을 따로 만들어** 함께 올린다(`lib/client-image.ts`).
  전시실 액자는 썸네일, 원본은 눌러서 상세를 볼 때만 받는다.

- 옛 작품은 `node scripts/backfill-thumbnails.mjs --apply` 로 따라잡는다(여러 번 돌려도 안전).
- **Storage 주소 파싱은 `lib/storage-path.ts` 를 쓴다.** 직접 정규식을 쓰지 말 것 —
  `firebasestorage.googleapis.com` 이 `storage.googleapis.com` 을 문자열로 포함해서,
  순서를 잘못 잡으면 엉뚱한 경로가 나온다. 실제로 이 버그로 점검 스크립트가
  멀쩡한 파일을 고아로 잡을 뻔했다.

### 학교 상징·교표 (2026-07-20)

학교를 만들 때 AI가 그 학교를 웹에서 조사하고, 현관 위 동그란 자리에 교표를 건다.

**AI 조사로 되는 것과 안 되는 것이 갈린다 (프로덕션 실측):**
- 개교연도 → 찾는다(애월초 1923년). 부동산·학교정보 사이트에 있다.
- 교훈·교화·교목 → **거의 못 찾는다.** 학교 공식 홈페이지에만 있는데
  초등학교 홈페이지는 프레임/자바스크립트라 검색 도구가 본문을 못 읽는다.

그래서 **못 찾은 항목은 빈 칸으로 돌려주고 사람이 채운다.** 남의 학교 교화를
그럴듯하게 지어내면 그 학교 아이들이 틀린 걸 자기 학교 것으로 배운다.
API 가 `missing` 을 함께 돌려주고, 화면이 "이건 못 찾았으니 직접 적어주세요"라고 말한다.
이미 사람이 적어둔 칸은 조사 결과로 덮지 않는다.

- 조사 모델은 **`gpt-4.1`**. `gpt-4.1-mini` 는 검색을 한 번 하고 포기해서
  찾을 수 있는 개교연도까지 놓쳤다(실측). 학교당 한 번 부르는 호출이라 등급을 올렸다.
- **AI 교표는 진짜 교표가 아니다.** 실제 교표는 그 학교의 표식이라 흉내 낼 것이 아니고,
  검색으로 가져올 수도 없다. 교화·교목을 재료로 '우리 학교 마크'를 새로 그려주는 것이고,
  진짜 교표가 있으면 직접 올리는 버튼을 나란히 뒀다.
- 교표도 Storage 로 간다. 바꾸면 옛 파일을 지운다 — 안 지우면 계속 쌓인다.
- **고칠 수 있는 범위가 역할마다 다르다.** 이 학교 선생님은 학교 상징·교표만,
  총관리자는 전부. 교화·교표는 그 학교 선생님이 제일 잘 알고 틀려도 다시 고치면 되지만,
  학교 이름·위치·학년/반 수·대표 이미지는 담임 한 명이 바꾸면 학교 전체가 따라 바뀌고
  학년/반은 되돌릴 수도 없다. 서버(`PATCH /api/school`)가 `isSuper` 로 필드를 걸러내고,
  화면도 같은 선으로 칸을 숨긴다. **화면만 숨기고 서버를 안 막으면 막은 게 아니다.**
- 검색 결과는 **실행마다 다르다.** 같은 애월초를 두 번 조사했는데 한 번은 개교연도만,
  한 번은 교훈·교화·교목·출처 3개까지 나왔다. 못 찾았다고 한 번에 포기하지 말 것.
- **검증 스크립트가 진짜 학교 문서에 쓰면 안 된다.** verify-class-scope 가 애월초에
  '검증용 교훈'을 써놓고 지우지 않아 프로덕션에 그대로 남은 적이 있다.
  지금은 일회용 학교(`zz-scope-school`)를 만들어 거기에 쓰고 지운다.
- 검증: `BASE_URL=https://aweol.vercel.app node scripts/verify-school-profile.mjs`

### 학교 현관 로비 (2026-07-21)

현관문을 누르면 **로비 안으로 들어간다**(`/school/*/lobby`). 예전에는 창이 하나
뜨는 게 전부라 '들어간' 느낌이 없었다.

- 게시판 넷을 **한 벽에 몰지 않았다.** 정면에 학교 소개, 오른쪽에 공지와 건의함,
  왼쪽에 앨범. 다 몰면 들어서자마자 보여서 돌아다닐 이유가 없다.

### 학교 동물과 마주치기 (2026-07-21)

동물이 아바타를 알아본다. **여기도 AI 를 안 부른다** — 거리와 시간만 보면 되는 일이고,
아이가 앞에 서 있다고 매번 모델을 부를 이유가 없다.

- 부딪히면 밀려나며 "앗! 깜짝이야"
- 앞을 막고 서 있으면 2.5초 뒤부터 점점 서운해한다:
  "저기... 지나갈게요" → "잠시만 비켜주세요~" → "왜 자꾸 길을 막아요?" → "흥, 다른 길로 갈래요!"
- **말만 하고 끝나지 않는다.** 투덜거린 뒤 목표를 옆으로 틀어 **돌아간다.**
  계속 막혀 있으면 놀이가 아니라 답답한 일이 된다.
- 멀어지면 투덜거림이 처음으로 돌아간다 (다시 만나면 새 마음).

### 학교 현관 (2026-07-20)

학교 건물의 현관문을 누르면 '우리 학교' 창이 열린다. 반 교실이 '우리 반'이라면 여기는 학교 단위.

- **학교 소개** — 관리자 화면에서 채운 `profile` 을 그대로 읽어 보여준다.
  여기서 또 입력받으면 같은 값이 두 군데 생긴다.
- **공지**(`schools/*/hallNotices`) — 학교 전체가 읽고, 그 학교 교직원이 쓴다.
  반 알림장과 다른 것이다.
- **건의함**(`schools/*/suggestions`) — **공개가 아니다.** 낸 본인과 그 학교
  교직원만 읽는다. 전교생이 다 보는 게시판이면 아이가 정작 하고 싶은 말을 못 쓴다.
  그래서 질의도 갈라야 한다 — 교직원은 전체, 아이는 `where(authorUid == 내 uid)`.
  아이가 전체 조회를 하면 규칙에 막혀 그냥 실패한다.
  답변은 교직원만 달고, 낼 때 `reply` 를 미리 채워 자문자답하는 것도 막았다.
- **앨범** — 사진을 따로 올리지 않는다. 이미 올라온 작품의 **썸네일**을 모아 보여준다.
  새로 올리게 하면 저장 용량이 또 늘어난다.
  다만 작품에 `schoolId` 가 없어서 반을 돌며 모은다. 반이 많아지면 질의가 늘어나니
  그때는 작품에 `schoolId` 를 심고 collectionGroup 한 번으로 바꿔야 한다.
- 검증: `node scripts/verify-hall.mjs` (규칙 검증이라 배포 불필요)

**규칙 파일 함정:** `/** ... */` 주석 안에 `classes/*/notices` 처럼 `*/` 가 들어가면
주석이 거기서 닫혀 그 뒤가 코드로 읽힌다. 배포는 본문 없는 400 만 돌려주므로
원인을 찾기 어렵다. 주석에 경로를 쓸 때 `*` 를 피할 것.

### 학교 동물 (2026-07-20)

운동장에 학교당 한 마리(`schools` 아래 `pet/main`). 모두가 함께 먹이고 물 주고 쓰다듬는다.

- **배고픔·목마름을 숫자로 저장하지 않는다.** 마지막으로 챙겨준 *시각*만 두고
  화면에서 '몇 시간 지났나'로 계산한다. 수치를 저장하면 시간마다 깎아줄 서버가
  필요한데 그건 학교 수만큼 도는 배치다. 시각 하나면 아무도 안 볼 때도 배가 고파진다.
- **위치도 저장하지 않는다.** 동물이 어슬렁거리는 건 화면 안에서만 산다.
  좌표를 저장하면 아이가 볼 때마다 쓰기가 생겨 학교당 하루 수천 번이 된다.
- **말은 AI가 아니다.** 상태에 맞는 문장을 골라 준다. 25명이 쉬는 시간마다 말을 걸면
  그게 다 요금이고, 아이가 쓴 말을 그대로 모델에 보내는 것도 내키지 않는다.
- 규칙이 두 가지를 잡는다: 이름·종류는 교직원만, `careCount` 는 **한 번에 1씩만**.
  안 막으면 9999를 써넣고 1등이 된다.
- 화면에서도 충분히 채워진 항목은 버튼이 꺼진다 — 배부른데 또 먹이는 건 놀이가 아니고,
  열어두면 연타로 쓰기가 폭발한다.
- 검증: `node scripts/verify-pet.mjs`

**`*/` 함정을 또 밟았다.** 이번엔 TS 주석에 `` `schools/*/pet/main` `` 이라고 써서
주석이 일찍 닫히고 "Unterminated template literal" 이 났다. firestore.rules 에서 겪은
것과 같은 실수다. **주석에 경로를 쓸 때 `*` 를 넣지 말 것.**
찾는 법: `grep -rn '\*/[a-zA-Z{]' src firestore.rules`

### 운동장 달리기 (2026-07-20)

타원 트랙 한 바퀴. **흰 선을 밟거나 안쪽으로 질러가면 탈락.**

- 트랙 계산은 `lib/track.ts` **한 곳**에만 둔다. 그림과 판정이 같은 숫자를 봐야 한다 —
  어긋나면 아이는 안 밟았는데 탈락한다. 흰 선도 판정선(`LANE_HALF`) 자리에 그린다.
- **가로지르기는 거리만으로 못 막는다.** 트랙 한가운데를 질러가도 중심선 근처라
  통과해버린다. 그래서 체크포인트 8개를 **순서대로** 지나야 한 바퀴로 친다.
  (출발선 앞에서 앞뒤로 왔다갔다해도 0바퀴인지 검증에 넣어뒀다)
- **시간은 서버가 잰다**(틀린그림과 같은 원칙). 출발할 때 서버가 시각을 찍고
  도착할 때 뺀다. 12초 미만·10분 초과는 기록으로 안 남긴다.
  같은 판으로 두 번 도착하는 것도 막는다(409).
- 기록은 사람당 한 줄(`trackRecords/{uid}`)에 자기 최고만. 순위표를 읽어도
  학생 수만큼만 읽는다. **클라이언트 쓰기는 규칙으로 완전히 막았다** —
  서버가 재는데 클라이언트가 쓸 수 있으면 재는 의미가 없다.
- 아바타는 학교와 같은 `WalkerAvatar` 를 쓰고 `TrackJudge` 가 위치를 읽기만 한다.
  달리기용 아바타를 따로 만들면 조작감이 달라지고 코드가 두 벌이 된다.
- 출발선으로 되돌리는 건 **카운트다운이 시작될 때**다. '출발!' 에 옮기면
  세는 동안 딴 데 서 있다가 갑자기 순간이동한다.
- 검증: `node --experimental-strip-types scripts/verify-track-math.mjs` (판정 계산)
  + `BASE_URL=https://aweol.vercel.app node scripts/verify-track.mjs` (기록 위조)

**검증 스크립트가 판정 코드를 베껴 쓰지 않는다.** `verify-track-math.mjs` 는
`--experimental-strip-types` 로 `src/lib/track.ts` 를 그대로 불러온다.
베껴 두면 화면은 고쳤는데 검증은 옛 숫자를 보게 된다.

### 기억창고 · 교사 포트폴리오 (2026-07-20)

한 해가 끝나면 반을 갈무리한다. 졸업한 뒤에도 자기 반을 볼 수 있다.

**Firestore 에 그대로 두지 않는 이유가 요금이다.** 한 반에 작품·숙제·제출물·낙서가
수백 건인데, 졸업한 반이 쌓일수록 목록 한 번 여는 읽기가 계속 는다.
그래서 전체는 **Storage 에 JSON 한 덩어리**로 넣고(`archives/{schoolId}/{year}-{classId}.json`)
Firestore 에는 요약 한 줄만 남긴다. 기억창고를 열면 **문서 1건 + 파일 1개**다.

**갈무리 파일은 주소만 알면 누구나 받는다.** 그래서 지금 규칙으로 가려진 것을
넣으면 그 순간 가림막이 사라진다. 담지 않는 것:
- **명부** — 아이 이름 목록. 몇 명이었는지 숫자만 남긴다.
- **'선생님만 보기' 제출물** — `publicToClass` 인 것만 담는다(이건 지금도 공개다).
- **퀴즈 문항·정답** — 파일 주소만 알면 답을 다 보게 된다. 제목과 개수만.

`verify-archive.mjs` 가 이 셋을 **실제 파일 내용으로** 확인한다
(문자열이 안 들어 있는지 직접 본다).

- **원본은 지우지 않는다.** 아이들이 만든 걸 앱이 알아서 없애면 안 된다.
  반에 `isArchived` 만 세워 활성 목록에서 빠지게 한다. 진짜 삭제는 사람이 결정할 일.
- 갈무리는 **담임(과 총관리자)만**. 남의 반을 치울 수 없다.
- 포트폴리오는 `collectionGroup('archives')` 를 `archivedBy` 로 거른다.
  **collectionGroup 은 중첩 규칙이 안 먹으니 `match /{path=**}/archives/{id}` 를 따로 뒀다**
  (이 프로젝트에서 반복해서 밟는 함정).
- 검증: `BASE_URL=https://aweol.vercel.app node scripts/verify-archive.mjs`

### 다중접속 (2026-07-21)

같은 학교·같은 교실에 있는 친구가 실제로 보인다. 이름표가 머리 위에 뜬다.

- **영속 데이터는 Firestore, 휘발성 위치는 RTDB.** 위치를 Firestore 에 쓰면
  25명 40분 수업에 문서 쓰기가 수십만 번이라 한 번에 $10 쯤 나온다.
  1초 뒤면 의미가 없어지는 값을 문서로 남길 이유도 없다.
- 방은 공간마다 따로다(`rooms/{schoolId}/{roomKey}`). `school` / `class-3-1` / `lobby`.
  **경로가 달라서 다른 공간 사람은 애초에 안 받는다** — 받아놓고 거르면
  안 볼 사람 몫까지 내려받는 셈이다.

**요금을 잡는 네 가지 (실측):**
1. 초당 5번만 보낸다 (60fps 로 보내면 12배).
2. 움직일 때만 보낸다 — 아이들은 대부분 서서 구경한다.
3. 좌표는 문자열 하나로 자른다 (`"12.34,5.67,1.23"`).
4. **이름·아바타는 들어올 때 한 번만.** 매 틱에 실으면 89바이트, 위치만이면 40바이트 —
   **2.2배 차이**고 무료 한도(10GB/월) 기준 수업 67회와 149회의 차이다.

- 끊겼다 붙으면 `onDisconnect` 가 이미 자리를 지운 뒤라, 위치만 갱신하면
  **이름 없는 유령**이 된다. `.info/connected` 를 보고 다시 붙을 때 전체를 새로 쓴다.
- 받은 좌표로 곧장 옮기지 않는다. 초당 5번이라 그대로 쓰면 뚝뚝 끊긴다 —
  마지막 자리를 향해 부드럽게 따라가게 해서 사이를 메운다. 이게 5Hz 를 안 티나게 만든다.
- 규칙(`database.rules.json`)이 **남의 아바타를 못 옮기게** 막는다. 정해진 칸
  외에는 못 쓰고, 이름·좌표 길이도 제한한다.
- 배포는 `node scripts/deploy-rtdb-rules.mjs` — **Firestore 와 스코프가 다르다.**
  `cloud-platform` 만으로는 401 이고 `firebase.database` + `userinfo.email` 이 있어야 한다.
- **숨소식(heartbeat)은 화면 그리기와 따로 돈다.** 처음에는 `useFrame` 에서만
  보냈는데, 탭이 뒤로 가거나 화면이 안 그려지면 `requestAnimationFrame` 이 멈춰
  숨소식도 같이 멈췄다. 그러면 접속해 있는 사람이 남의 화면에서 사라졌다가
  다시 보면 나타난다 — **'깜빡였다 사라진다' 는 증상이 이것이었다.**
  지금은 `setInterval` 로 따로 돌린다.
- 사라짐 판정은 45초, 숨소식은 8초. 판정이 짧으면 가만히 선 사람이 숨소식 사이에 사라진다.
- 시각 비교는 `.info/serverTimeOffset` 을 더해 **서버 시각으로** 한다.
  (실측해보니 이 컴퓨터는 154ms 밖에 안 어긋나 있어서 깜빡임의 원인은 아니었지만,
  시계가 크게 틀어진 기기에서는 같은 증상이 난다)
- 검증: `node scripts/verify-presence.mjs`

### 술래잡기 · 마을 (2026-07-21)

다중접속 위에 올린 첫 놀이 둘. **위치는 presence 가 이미 나르므로 둘 다
자기 것만 다룬다** — 술래잡기는 '누가 술래인가', 마을은 아무것도.

**술래잡기** (`/school/*/class/*/tag`) — 반 단위 방, 3분 한 판.
- **판정은 술래의 화면이 한다.** 서버가 하려면 모든 아이 좌표를 서버가 다시
  계산해야 하는데 그건 지금 없는 서버를 하나 더 두는 일이다. 반 친구끼리
  이름 붙이고 하는 놀이라 여기까지가 맞다고 봤다.
- 대신 규칙이 막는다: **술래만 술래를 넘길 수 있다**(`it` 의 `.write` 가
  `data.val() === auth.uid`). 아무나 남을 술래로 만들면 놀이가 안 된다.
  점수도 본인 것만, 999 까지만.
- **술래 하나만 판정한다.** 모두가 서로를 보면 같은 순간에 여러 명이
  '내가 잡았다' 를 쓴다.
- 잡힌 직후 3초는 쉰다 — 안 그러면 술래가 핑퐁처럼 왔다갔다한다.
- 놀이터에 나무·바위·건초더미를 뒀다. 벌판이면 그냥 쫓기만 한다.
  술래는 조금 빠르다(5.4 vs 4.8) — 안 그러면 영영 못 잡는다.
- 검증: `node scripts/verify-tag.mjs`

**마을** (`/village`) — **새 기능이 아니라 입구를 모은 곳이다.**
학교·전시실·상점·지도가 건물로 서 있고 문을 누르면 들어간다.
메뉴를 누르는 것과 걸어가서 문을 여는 건 아이에게 다른 경험이다.
- 방은 `village` 하나 — 학교가 달라도 마을에서는 같이 있다.
- 길을 깔았다. 길이 없으면 어디로 가야 할지 모른다.

### 놀이와 학습 잇기 (2026-07-21)

숙제 검사 → 도장 → **놀이 아이템** → 술래잡기. 끊겨 있던 마지막 칸을 이었다.

**"숙제를 낸 아이만 쓸 수 있게" 하지 않았다.** 집 사정으로 숙제를 못 하는 아이가
놀이에서까지 밀리면 안 된다. 놀이는 누구나 그냥 할 수 있고, 아이템은 얹어지는
재미다. 화면에도 "없어도 노는 데는 아무 지장 없어요" 라고 적어뒀다.
값도 싸게 잡았다(숙제 두세 번이면 산다).

- **소모품이라는 개념이 새로 생겼다.** 꾸미기는 한 번 사면 계속 갖지만
  놀이 아이템은 쓰면 없어진다. 그래서 **도장을 계속 쓸 데가 생긴다** —
  지금까지는 꾸미기 몇 개 사고 나면 도장이 쌓이기만 했다.
- **쓰는 것도 서버가 깎는다.** 화면에서만 세면 안 가진 아이도 계속 쓸 수 있고,
  그러면 산 아이만 손해다. 한 사람이 20개까지만 쟁일 수 있다.
- 🥾 **바람의 신발** — 한 판 동안 조금 빠르다. 다만 술래보다 훨씬 빨라지지는
  않게 했다(+0.8). 아이템 하나로 아무도 못 잡으면 나머지가 재미없어진다.
- 🛡️ **튼튼 방패** — 잡히면 한 번 되돌려준다.
  **규칙을 우회하지 않는다** — 잡힌 순간 나는 술래가 되고, 술래는 술래를 넘길 수
  있으므로(규칙), 나를 잡은 사람에게 곧장 돌려주는 것으로 구현했다.
  남의 화면이나 남의 데이터에 손대지 않는다.
- 🩹 **구름 신발** (달리기) — 선을 한 번 밟아도 봐준다.
  **달리기에는 빨라지는 아이템을 안 넣었다.** 순위표가 걸린 놀이라
  아이템을 산 아이가 빨라지면 기록이 의미를 잃는다. **시간에는 손대지 않는다.**
- 🔍 **돋보기** (틀린그림) — 못 찾은 곳 하나를 알려준다.
  정답 좌표는 애초에 화면으로 안 내려가므로 **서버가 골라준다.**
  그리고 **쓴 횟수를 기록에 남긴다**(`hints`) — 순위표에 돋보기 쓴 기록과
  안 쓴 기록이 섞이면 순위가 의미를 잃는다.
- 검증: `BASE_URL=... node scripts/verify-play-items.mjs`
  + `verify-play-items2.mjs` (돋보기·구름 신발)

**함정:** 틀린그림에서 **학생 액션은 `PUT`, 교사 출제는 `POST`** 다.
검증을 POST 로 짰다가 전부 403 을 맞았다.

### 교실 복도 — 계단으로 찾아가기 (2026-07-21)

현관에서 계단을 올라 복도를 걷고 문패를 보고 우리 반에 들어간다.
(`/school/*/floor/[floor]`)

**배치가 반 수에 따라 저절로 늘어난다.** 손으로 고칠 것이 없다.
- **한 층에 두 학년**, 복도를 사이에 두고 왼쪽·오른쪽.
  1층 = 1·2학년, 2층 = 3·4학년, 3층 = 5·6학년.
- 반이 늘면 복도가 길어지고(문 간격 3.2m), 학년이 늘면 층이 생긴다.
- 실측: 6학년×6반 → **3층, 한쪽에 6칸, 복도 25m**. 6학년×12반이면 44m.
- `floorOfGrade` / `gradesOnFloor` / `floorCount` 세 함수가 규칙 전부다.

- **내 반 문은 다르게 보인다** — 문틀·문이 금색이고 문패에 ⭐ 가 붙는다.
  다른 층이면 "우리 반은 N층이에요 — 가기" 버튼이 뜬다.
- 다중접속 방은 **층마다 따로**(`floor-{n}`). 3층 복도에 1층 아이가 서 있으면 안 된다.

### `viewport-fit=cover` 없이 안전영역을 쓰면 죽은 코드다 (2026-07-21)

`env(safe-area-inset-*)` 은 `<meta name="viewport" ... viewport-fit=cover>` 가
있어야 값이 나온다. **없으면 항상 0** 이고, CSS 는 오류 없이 조용히 아무 일도
안 한다. 실제로 `.pos-top-safe` 를 만들어놓고 아이폰에서 효과가 0 이었다.
Next 에서는 `layout.tsx` 의 `export const viewport` 에 `viewportFit: 'cover'`.

확인법 — 빌드 산출물에서 직접 본다:
```
curl -s <배포주소> | grep -o '<meta name="viewport"[^>]*>'
```

곁들여: 아이폰 사파리는 **주소창이 위에 있으면 `inset-top` 이 0** 이다.
그래서 위가 잘리던 진짜 원인은 안전영역이 아니라 `h-screen`(100vh 가 주소창
뒤로 넘어감)이었고, 고친 건 `h-dvh` 다. 안전영역은 가로모드·홈바용이다.

  친구와 같은 판을 못 보고, 검증도 못 한다.
- **점수는 시간이 아니라 뒤집은 횟수.** 시간으로 재면 손 빠른 아이가 유리하고
  아이가 화면을 급하게 넘긴다. 기억해서 맞히는 게 목적이다.

### 양궁 — 순위표가 걸린 게임은 서버가 채점한다 (2026-07-21)

점수를 클라이언트가 보내면 아무 숫자나 적어 보낼 수 있다. 그래서 두 단계다.

1. 시작할 때 **서버가 씨앗을 정해 준다** (흔들림·바람이 여기서 나온다)
2. 낼 때 아이는 **'언제 쏘았는지'(ms) 만** 보내고, 서버가 그 씨앗으로 조준점을

**계산에서 잡은 진짜 버그 둘:**
- 한가운데가 **11점**이었다. `ceil(0)=0` 이라 `11-0=11`. 만점을 넘는다.
- **바람이 씨앗과 무관하게 거의 고정**이었다. 씨앗 20개의 첫 화살 바람이 전부
  7.0 — 한 번 외우면 매번 같은 곳을 노리면 된다. xorshift 를 얕게 돌린 탓이라
  곱셈 기반 해시로 바꿨다. **씨앗을 쓰는 곳에서는 흩어짐을 꼭 재볼 것.**

- 계산 검증: `node --experimental-strip-types scripts/verify-archery.mjs` (27건)

### 전시관 — '나이스에 없는 학교' 일 뿐이다 (2026-07-22)

개인 전시장·박물관 용도. **구조를 새로 만들지 않았다.** 반(class)도 활동도
경로도 규칙도 학교와 똑같다. 처음에 새 구조를 짜려다가, 그럴 필요가 없다는
지적을 받고 다시 잡았다 — 손댈 데가 열 배 차이였다.

바뀐 것은 **보이는 것 둘뿐**이다.

- `schools/{id}.kind === 'gallery'` 면 창문 문패 대신 **건물에 걸린 큰 배너**.
  작은 문패로는 전시 주제가 안 읽히고 '학교의 반' 처럼 보인다. 4개까지만 건다.
- 배너를 누르면 **교실을 건너뛰고** `.../class/{id}/exhibits` 로 간다.
- 반 이름은 `displayName`(전시 주제)이 있으면 그걸 쓴다. 없으면 '3-1'.
  **학년·반 번호는 그대로 둔다** — 경로도 규칙도 그걸 쓴다.

기존 학교 문서는 손댈 필요가 없다(`kind` 가 없으면 'school').

**화면에서 만든다:** 학교 설정에 '🏫 학교 / 🎨 전시관' 고르기(총관리자만),
반 손보기에 '전시 주제' 칸(전시관일 때만). `kind` 는 **API 에서도** 총관리자만
바꾸게 막았다 — 담임이 자기 학교를 전시관으로 바꾸면 다른 반 선생님들이
교실을 못 찾는다.

- 검증(배포본): `BASE_URL=... node scripts/verify-gallery-kind.mjs`

**배포 검증할 때 주의:** 커밋 직후 40~50초로는 모자랄 때가 있다. 이번에
앞의 세 건이 옛 빌드를 쳐서 400 이 났고, 네 번째부터 새 빌드였다.
**실패가 나면 배포가 끝났는지부터 보고 다시 돌릴 것.**

- 처음 2초는 안 잡는다(시작하자마자 잡히면 억울하다)
- **기록은 랭킹에 안 올린다.** 진짜 친구와 논 것과 섞이면 안 된다

### 3D 경기장은 화면을 가득 채운다 (2026-07-22)

### 양궁 — 흔들림은 중앙을 지나야 요령이 생긴다 (2026-07-22)

조준점이 리사주 곡선(x·y 따로 흔들림)이라 **정중앙을 한 번도 안 지났다.**
중앙 대칭이긴 했지만 만점을 노릴 타이밍이 없어 감으로만 쏘게 됐다.

**한 직선 위를 오가게** 바꿨다(`aimAt` = 방향 × sin). sin 이 0 일 때마다
정중앙을 지나므로, 그 순간 쏘면 만점이다 — 요령이 생긴다. 직선의 기울기는
화살마다 달라(`angle`) 늘 같은 자리를 노릴 순 없다. `swayX/swayY` → `reach/angle`.

바람은 그대로 landing 을 민다. 겨눈 자리(십자선)를 보고 **바람 반대쪽으로
살짝 옮겨 쏘는 게** 이 게임의 요령이다.

**화면도 고쳤다:** 활이 과녁 앞에 거대하게 떠 있었다(ㅋㅋ). 카메라 앞
왼쪽 아래로 내려 손에 든 것처럼 했고, 활은 손떨림 정도만 흔들린다.
그리고 **조준 십자선(`Reticle`)을 과녁에 넣었다** — 어디를 겨누는지 안 보이면
요령이 안 생긴다. 이게 원래 없던 게 진짜 문제였다.

- 검증: `scripts/verify-archery.mjs` — 정중앙을 4초에 4번 지남, 한 직선 위,
  화살마다 기울기 다름, 만점 가능(50/50)하되 연타로는 드묾(3/286).

### 양궁 대결 — 턴제 1:1 (2026-07-22)

둘이 각자 기계로 들어와 번갈아 쏜다. 방 만들면 4자리 번호 → 상대가 그 번호로
참가. 1번 쏘면 2번 차례, 서로 상대가 쏘는 걸 실시간으로 본다.

- **점수·차례는 서버가 정한다.** 방 문서에 씨앗 하나 — 둘 다 같은 화살 번호에서
  같은 바람·흔들림(공평). 각자 쏜 발 수가 곧 화살 번호.
- 격발 **15초 제한.** 서버가 `turnStartedMs` 부터 다시 재서 넘겼으면 0점,
  차례가 넘어간다. 화면은 15초 카운트다운을 돌리고 넘으면 자동 격발(0점).
- 방 문서는 읽기는 로그인하면(상대 화면을 같이 본다), 쓰기는 서버만.
  대결은 순위표에 안 올린다 — 씨앗을 봐도 상대도 똑같이 유리하다(혼자 판과 다름).
- 화면은 방 문서를 `onSnapshot` 으로 읽어 따라간다. 내 차례일 때만 활·조준선이
  뜨고, 상대 차례면 상대 자국을 보며 기다린다.

턴 로직은 `archery-duel.ts` 에 계산만: `whoseTurn`·`isDone`·`duelResult`.
- 검증: 턴 20건(`verify-archery-duel.mjs`) + 배포본 API 흐름

**2:2 는 `size` 만 늘리면 되게 뒀다.** 팀 합산·짝 배치는 아직 안 함(다음).

### 양궁 난이도 — 타이밍 정밀도로 갈린다 (2026-07-22)

`mul`(흔들림 빠르기)·`windMul`(바람 세기). 빠를수록 중앙을 지나는 순간이 짧아
타이밍이 빡세고, 바람이 셀수록 가운데서 쏴도 밀린다.

### 3D 화면이 80px 스크롤되고 있었다 (2026-07-23)

마을에서 **하단 메뉴 탭이 씹히고 안내 말풍선이 조이스틱·'타기' 버튼에 가려지던**
두 증상은 뿌리가 하나였다.

`(main)/layout.tsx` 가 하단 메뉴 자리로 `pb-20`(80px)을 더하는데, 그 안의
3D 페이지가 `min-h-dvh` 로 화면 전체를 차지하니 **문서가 정확히 80px 스크롤됐다.**
스크롤되면 씬 위의 버튼·조이스틱·말풍선(`absolute`)은 같이 밀려 올라가고
하단 메뉴(`fixed`)만 제자리라, 층층이 앉혀둔 것들(위 '화면 아래는 층이 셋이다')이
서로 겹친다. 아이폰에서는 그 스크롤 튕김 때문에 메뉴 탭까지 씹힌다.

증상으로 알아보는 법: **3D 와 하단 메뉴 사이에 흰 띠**가 보이고
좌상단 '← 지도로' 가 화면 위로 사라져 있으면 이것이다.

→ 씬 페이지 뿌리를 **`.scene-page`**(`position:absolute; inset:0`)로 바꿨다.
흐름에서 빠지면 높이가 0 이라 스크롤이 생길 수 없다. 9개 화면 전부.
브라우저 375×812 실측: 고치기 전 800/720(80px 스크롤) → 고친 뒤 720/720.
말풍선·순간이동·조이스틱·타기 네 상자가 서로 안 겹치는 것도 좌표로 확인했다.

**곁들여 잡은 것:**
- `RoleSwitcher`(🧪)가 `bottom: 5.5rem`(88px)로 박혀 있었다. 홈바 있는 아이폰의
  메뉴 높이(60 + 안전영역 34 = 94px)보다 낮아 **'더보기' 칸 위에 z-60 으로 올라앉아
  탭을 삼켰다.** 숫자를 박지 말고 `.pos-hint` 같은 층 클래스를 쓴다.
- 마을 '← 지도로' 가 `top-4` 였다(노치에 잘린다) → `.pos-top-safe`.

### drei `<Html>` 의 `pointerEvents="none"` 은 `center` 모드에서 안 먹는다 (2026-07-23)

`transform` 모드에서만 적용된다(drei 가 그 값을 `transformInner` 스타일에만 쓴다).
`center` 로 띄운 이름표는 **prop 을 줘도 탭을 삼킨다.** 마을 학교 간판은 학교 자리
바로 위에 뜨므로, "학교 자리를 누르면 들어가요" 가 간판을 누른 사람에게는 거짓말이었다.

→ `center` 인 곳은 `style={{ pointerEvents: 'none' }}` 로 직접 준다(14곳).
`transform` 인 곳은 prop 이 제대로 먹으므로 그대로 둔다.

찾는 법: `grep -rn 'center pointerEvents="none"' --include=*.tsx src` 가 비어 있어야 한다.

### 학교관리자 — 등급을 하나 끼워 넣었다 (2026-07-23)

권한이 넷에서 **다섯**이 됐다. `school_admin`(학교관리자)은 **한 학교 안의 중간관리자**다.

**왜 끼웠나.** 두 가지가 총관리자 한 사람에게 몰려 있었다.
1. **반 만들기가 아무 선생님에게나 열려 있었다.** 학년·반 구성은 학교가 정하는 것인데
   담임 한 명이 3-4 반을 임의로 만들면 실제 학교와 어긋난다.
2. **교사 승인이 전부 총관리자 몫이었다.** 학교가 늘면 감당이 안 되고, 애초에
   "이 사람이 우리 학교 선생님이 맞는가" 는 그 학교가 제일 잘 안다.

**누가 무엇을 하는가 (이 선을 흐리면 등급을 나눈 의미가 없다):**

| | 총관리자 | 학교관리자 | 선생님 |
|---|---|---|---|
| 학교 만들기 | ✅ | ❌ | ❌ |
| 학교관리자 임명 | ✅ | ❌ | ❌ |
| **반 만들기·지우기** | ✅ | ✅ | ❌ |
| **교사 승인** | ✅ | ✅ (우리 학교만) | ❌ |
| 반 안의 자료(숙제·퀴즈·명부) | ✅ | 맡은 반만 | 맡은 반만 |

- **학교관리자가 학교관리자를 임명할 수 없다.** 열어두면 한 번 뚫린 학교에서
  관리자가 계속 늘어난다. 임명은 총관리자만.
- **학교관리자도 반 안에서는 담임과 같은 선이다.** 학교 관리자라고 남의 반 숙제를
  고칠 수 있으면 안 된다 — 학교 단위 권한과 반 단위 권한은 다른 것이다.
  `classIds` 에 반이 있으면(겸직) 그 반에서만 담임이다.
- **학교관리자는 맡은 반이 없을 수 있다.** 그래서 신청할 때 학년·반을 묻지 않고
  승인 때 `classIds` 를 비운다. 교감·정보부장 같은 자리를 생각하면 된다.
- **반을 만든 사람이 담임이 되지 않는다.** 예전에는 선생님이 자기 반을 만들었으니
  만든 사람 = 담임이었다. 이제는 관리자가 학교의 반을 한꺼번에 세우므로,
  그대로 두면 **학교관리자가 온 학교의 담임**이 된다. 담임은 교사 승인 때 빈 반에 채워진다.

**users 문서를 딱 한 뼘 열었다.** 승인하려면 누가 신청했는지는 봐야 한다.
`pendingRole == 'teacher'` 이고 신청 학교가 내 학교인 문서만. 승인되면 pendingRole 이
지워지므로 **그 순간 다시 안 보인다.**
→ **질의 조건이 규칙과 정확히 같아야 한다.** 규칙이 열어준 것보다 넓게 물으면
문서 하나가 막히는 순간 질의 전체가 실패한다(`/admin/teachers` 참고).

**한 군데로 모은 것:** 화면 곳곳에 `role === 'teacher' || role === 'super_admin'` 이
흩어져 있었다. 등급이 늘 때마다 전부 찾아 고쳐야 하고 하나 빠뜨리면 **어떤 화면에서는
교직원이고 어떤 화면에서는 아닌** 상태가 된다. `auth-helpers.ts` 의 `isStaff` 로 모았다.
- `isStaff`(교직원) / `isSchoolManager`(학교 단위 관리자) / `canCreateClass` /
  `canApproveTeacher` — 서버 쪽 짝은 `firebase-admin.ts` 의
  `isStaffOfSchool` / `isSchoolAdminOfSchool` / `isStaffRole`.

**화면만 막으면 막은 게 아니다.** 담임에게 '+ 반 만들기' 를 보여주고 서버에서 403 을
주면 아무 설명 없이 실패한 것으로 보인다. 그래서 버튼을 감추고, 서버 문구도
"학교관리자 선생님께 요청해 주세요" 로 적었다.

- 검증: `BASE_URL=https://aweol.vercel.app node scripts/verify-school-admin.mjs`
  + `node scripts/verify-class-create.mjs` (일반 교사 403 · 담임 비어 있음까지 확인)
- **규칙을 고쳤으므로 `node scripts/deploy-firebase-rules.mjs` 를 돌려야 한다.**
- 첫 학교관리자는 신청할 사람이 승인할 사람을 겸할 수 없으니 **총관리자가 승인**한다
  (`/admin/teachers`).

### 전시실 공개 범위 · 갤러리 필터 (2026-07-23)

전시실마다 **학교 전체 / 우리 반만**을 고른다. 잠근 전시실은 전체 갤러리에서 빠지고,
그 반 사람(아이·담임·**학부모**)만 작품을 읽는다.

**화면에서만 숨기지 않았다.** 전체 갤러리는 작품을 `collectionGroup` 으로 긁는데
그 자리의 규칙(`match /{path=**}/artworks/{id}`)에서는 **경로를 쪼갤 수 없어**
어느 반 것인지 알 수 없다. 그래서 작품 문서가 `schoolId`·`classId`·`visibility` 를
함께 지닌다. 전시실 설정을 바꾸면 **그 안의 작품도 배치로 따라 고친다** —
안 고치면 "전시실은 잠갔는데 갤러리에는 그대로" 가 된다.

**갤러리**
- 학년 → 반 필터. 작품이 **있는** 학년·반만 칸을 만든다(누를 것 없는 칸은 방해다).
- **보관된 반의 작품은 빠진다.** 이것이 '해마다 리셋' 이다 — 갤러리에 연도 필드를
  따로 두면 해가 바뀔 때 누군가 전 작품을 손봐야 한다. 기억창고 갈무리는 이미 있는 절차다.
- 질의가 세 갈래다: 교직원(전부) / 학교 공개 / 내 반. **규칙보다 넓게 물으면
  Firestore 가 질의 전체를 거절해 갤러리가 통째로 빈다.**
- 전시실 화면도 같은 이유로 반 밖에서는 `visibility` 를 걸어서 묻는다.
  잠긴 전시실은 '거부' 가 아니라 **빈 결과**로 오므로, 화면이 전시실 문서를 보고
  "우리 반만 보는 전시예요" 라고 말해준다. 안 그러면 아이는 선생님이 아직 안 걸었다고 생각한다.

**밟은 함정 둘 — 둘 다 조용히 열려 있었다:**

1. **규칙의 `resource.data.get('키', 기본값) == ...` 는 질의 판정에 안 걸린다.**
   문서 하나를 읽을 때는 제대로 막는데, **질의는 그냥 통과시킨다**(실측: 잠긴 작품이
   비로그인 collectionGroup 조회에 그대로 딸려 왔다). 기본값 있는 `get()` 을
   질의의 `where` 와 맞춰볼 수 없기 때문으로 보인다.
   → **필드를 직접 비교한다.** 대신 필드가 없는 문서는 평가가 실패해 막히므로
   `scripts/backfill-artwork-scope.mjs` 를 **규칙 배포보다 먼저** 돌려야 한다.
   찾는 법: `grep -n "data.get(" firestore.rules` — 질의에 쓰이는 자리인지 본다.

2. **학부모는 `classIds` 가 아니라 `children` 에 반이 있다.**
   규칙은 객체 배열에서 `classId` 만 뽑아낼 수 없다. 그대로 뒀으면 '우리 반만' 에서
   **학부모만 자녀 전시를 못 봤다.** 서버(`/api/student-code`)가 `childClassIds` 를
   평평하게 적어주고 규칙이 그걸 본다(`inMyClasses`). `classIds` 에 그냥 넣지 않은
   이유는 그게 '내가 속한 반' 이라 아이·교사용 판정에 함께 걸리기 때문이다.
   옛 계정은 `scripts/backfill-parent-classes.mjs`.

**검증 스크립트가 실패 이유를 뭉뚱그리면 안 된다.** 복합 인덱스가 만들어지는 동안에는
질의가 `failed-precondition` 으로 실패하는데, 그걸 '거부됨' 으로 세면
"규칙이 잘 막고 있다" 는 **거짓 통과**가 나온다. 실제로 처음 돌렸을 때 그 상태였다.
그리고 **검증은 화면과 똑같이 물어야 한다** — 다르게 물으면 멀쩡한 화면을
고장났다고 하거나 그 반대가 된다.

- 검증: `node scripts/verify-exhibit-visibility.mjs` (21건)
- 배포 순서: **backfill → 규칙 배포 → 코드 푸시.** 규칙이 먼저 조여지면
  옛 코드의 갤러리 조회가 막힌다(잠깐이라도 갤러리가 빈다).
