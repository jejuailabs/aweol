# 마을을 실제 지도와 잇기 (설계안)

2026-07-21 조사. 아직 구현 전이다.

## 무엇을 풀려는 건가

지금 마을(`/village`)은 손으로 배치한 건물 4채짜리 52m×52m 고정 맵이다.
모든 학교가 똑같이 생겼고, 걸어다닐 것이 없다.

학교 문서에는 이미 **실제 좌표**(`lat`/`lng`)가 있다. 이걸 써서
**자기 학교 동네**를 걸어다닐 수 있게 만드는 것이 목표다.

## 참고한 것과, 왜 그대로 안 쓰는가

### Bruno Simon 포트폴리오 (bruno-simon.com)

기술을 베끼려고 본 게 아니라 **'꽤 넓은 구간을 게임처럼 돌아다니고, 지도로 워프한다'**
는 감각을 보려고 봤다. 실제 지도는 안 쓴다 — Blender 로 손수 만든 세계고,
보이는 모델과 별개로 물리용 단순 도형을 따로 둬서 계산을 줄인다.
전체 2MB 미만, 조명과 그림자가 아예 없다.

→ **가져올 것:** 조명·그림자를 줄이는 방식, 보이는 것과 부딪히는 것을 나누기,
   그리고 **한 덩어리 세계가 아니라 구간 + 워프**.
→ **안 맞는 것:** 학교마다 동네를 손으로 만들 수는 없다.

### 그래서 구조는 '구간 + 워프' 다

800m 를 이어 붙인 한 덩어리로 만들 필요가 없다. 그렇게 하면 빈 땅을 오래 걷고,
받아야 할 것도 늘고, 아이가 길을 잃는다.

대신 **지도 화면이 워프 허브**가 된다:

```
지도(실제 지도 위) ──워프──> 학교 앞      (지금 SchoolScene)
                  ──워프──> 마을 상가     (구간)
                  ──워프──> 바닷가·항구   (구간)
                  ──워프──> 오름·공원     (구간)
```

각 구간은 **80~150m 짜리 작은 무대**다. 그 안에서는 마음껏 돌아다니되
경계에 서면 "여기서 나가기" 가 뜨고 지도로 돌아온다.
구간 하나는 지금 마을(52m)과 놀이터(26m) 사이 크기라 이미 검증된 규모다.

이 방식의 이점:
- 받는 양이 구간 단위로 쪼개진다 (한 번에 2.2KB 전부가 아니라 필요한 구간만)
- 빈 땅을 안 만든다 — 볼 것이 있는 곳만 무대로 만든다
- 구간마다 성격을 줄 수 있다 (상가는 상점, 바닷가는 놀이, 오름은 달리기)
- **다중접속 방을 구간별로 나눈다** — 한 방에 다 몰리지 않는다

### 지도 API 를 실시간으로 부르기 (Mapbox 등)

가장 쉬워 보이지만 **이 프로젝트에는 안 맞는다.** 아이가 마을에 들어올 때마다
지도 API 를 부르면 아이 수만큼 요금이 붙는다. 지금까지 지켜온 원칙과 정면으로 어긋난다.

그리고 **정책상으로도 안 된다.** OSM 계열 무료 서비스는
"앱에서 주기적으로 보내는 요청은 대량 조회로 보며 강력히 권장하지 않는다",
"결과는 반드시 로컬에 캐시하라"고 못박고 있다. 실시간 호출은 서비스 이용 정지 사유다.

## 안: **학교를 만들 때 한 번 굽는다**

기억창고와 같은 패턴이다 — **Firestore 에 수백 건 대신, Storage 에 파일 하나.**

```
학교 생성/설정 →  Overpass 로 그 좌표 주변 400m 조회 (한 번)
              →  걸어다닐 좌표계(미터)로 변환·단순화·반경 밖 잘라내기
              →  Storage 에 village-{schoolId}.json
              →  아이는 이 파일 하나만 받는다. 지도 API 를 아예 안 부른다.
```

### 실측 (애월초 33.4626, 126.3316 · 반경 400m)

| | |
|---|---|
| 건물 | 14채 (애월읍사무소, 애월체육관, 애월초등학교, 애월중학교 …) |
| 길 | 30조각 |
| 물·공원 | 2 (한담해변 산책로 등) |
| **파일 크기** | **6.1KB → gzip 2.2KB** |
| 걸어다닐 크기 | 791m × 790m |
| 가로지르는 시간 | 약 158초 (아바타 5m/s) |

**작품 썸네일 하나가 보통 40KB 다.** 마을 하나가 그 1/20 이다.
학교가 100곳이어도 전부 합쳐 600KB 다.

### 왜 이 방식이 맞나

- **요금:** 아이가 몇 명이 들어오든 지도 API 호출은 0. 학교당 딱 한 번.
- **정책:** 대량·반복 호출이 아니라 학교 생성 시 1회 + 로컬 캐시 — 정책이 허용하는 형태.
- **속도:** 2.2KB 는 어떤 회선에서도 즉시 받는다.
- **일관성:** 이미 쓰는 패턴이다(기억창고 JSON, 썸네일). 새 개념이 없다.

### 잘라내기가 꼭 필요하다

Overpass 는 반경에 **걸친 길 전체**를 준다. 안 자르면 7km 짜리 도로가 딸려와
마을이 7138m × 2819m 가 된다(실측). 반경 밖 구간을 끊어내면 791m × 790m 가 된다.

## 남은 결정 (사람이 정할 것)

1. **반경** — 400m 는 2분 반이면 가로지른다. 800m 면 넉넉하지만 빈 땅이 늘어난다.
2. **높이 정보** — OSM 에 층수가 없는 건물이 많다. 지금은 전부 6m 로 뒀다.
   동네 사진을 보고 어림잡거나, 학교가 직접 고치게 할 수도 있다.
3. **사유지 문제** — 남의 집이 그대로 3D 로 서 있는 게 괜찮은가.
   건물 모양은 공개 데이터지만, **아이들이 노는 공간에 이웃집이 재현되는 것**은
   기술 문제가 아니라 정서 문제다. 이름 있는 공공시설만 세우고 나머지는
   뭉뚱그린 블록으로 두는 쪽이 안전해 보인다.
4. **다중접속 방 구분** — 지금 마을은 `village` 방 하나다. 학교별 동네가 생기면
   `village-{schoolId}` 로 나눠야 자기 동네 친구를 만난다.

## 구현 순서 (제안)

1. `/api/village` — 좌표를 받아 Overpass 조회 → 변환 → Storage 저장 (총관리자/교사만)
2. `VillageScene` 이 고정 배치 대신 JSON 을 읽어 그리게
3. 건물에 이름표, 학교 건물은 눌러서 들어가게 (지금 문 누르는 것과 같게)
4. 다중접속 방을 `village-{schoolId}` 로
5. 검증: 파일 크기 상한, 반경 밖 좌표가 없는지, 사유지 필터가 실제로 걸리는지
